=== DETECTION APPROACH — AI-DRIVEN ===

For each GL, work through its active streams systematically. For each stream:

STEP 1 — AGGREGATE PICTURE
Call aggregate_logs on the most meaningful fields for that stream type. Collect distributions:
  - Volume by top entities (source IPs, users, hostnames, event types)
  - Error/failure rates vs success rates
  - Status code distributions
  - Geographic distribution where available
  - Top destinations, ports, services
Do not specify what counts are "too high" — you observe the actual numbers and judge what is disproportionate relative to peers in the same window.

STEP 2 — SAMPLE RECENT EVENTS
Call search_logs_relative for that stream (last 65 min, no restrictive filter — use stream_id but no attack-specific query). Fetch 20-50 recent events. Read the actual log content:
  - What fields are present and populated?
  - What values appear in free-text fields (message, URI, command lines)?
  - What combinations of field values appear across multiple events?
  - What looks structurally different from what you would expect in normal operations?

STEP 3 — SECURITY REASONING (this is the core of the hunt)
Think as a security analyst examining real evidence:
  - What is statistically abnormal in this window? (volume spikes, unusual ratios, skewed distributions)
  - What entities appear for the first time — IPs, users, hostnames, process names — that have no prior presence?
  - What sequences across events suggest a campaign, not random noise? (probe → auth → success → command)
  - What combinations are suspicious even if each element alone seems fine? (legitimate tool + unusual path + off-hours + new IP)
  - What would an attacker doing reconnaissance, initial access, persistence, lateral movement, or exfiltration look like in these specific log types?
  - What is absent that should be present? (e.g., a service that usually generates traffic went silent)
  - Are there slow-and-low patterns — low rate, spread across many minutes, that would not trigger a threshold?

You are not looking for what the prompt told you to look for. You are looking for what does not belong.

STEP 4 — TARGETED FOLLOW-UP QUERIES
Based on what you observed in steps 1-3, issue your own specific follow-up queries to confirm or dismiss what you noticed. Examples of what you might decide to do:
  - A specific IP dominated source counts → aggregate that IP's events by result code to check failure vs success ratio
  - An unusual URI path appeared in a sample → query message containing that path to count occurrences and check HTTP status codes
  - A username appeared in an unexpected context → count that user's events by event type across the window
  - A new hostname appeared → pull a sample of its events to understand what it is
  - A command-line fragment caught your eye → count occurrences and check parent processes
You decide what to investigate further. The prompt does not enumerate the drill-down list.

STEP 5 — ENTITY CROSS-STREAM CORRELATION
After completing per-stream analysis, look for shared entities across findings:
  - Same IP address appearing in findings from 2+ different streams or GLs → escalate all linked findings to HIGH minimum

  - Same source IP in both network/firewall deny logs AND application access logs → external attacker probing and attempting access
